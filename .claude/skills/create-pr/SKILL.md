# Create PR (develop → main)

## 목표

develop 브랜치의 변경사항을 분석하여 PR description을 생성하고, `gh` CLI로 main 브랜치를 향한 PR을 **새로 생성하거나 기존 열린 PR을 갱신**합니다.
사용자는 결과 PR URL만 확인하면 됩니다.

## 사전 조건 검증 (필수, 순서대로 실행)

작성을 시작하기 전에 아래를 반드시 확인하세요. 하나라도 실패하면 중단하고 사용자에게 안내합니다.

1. **현재 브랜치 확인** — `git branch --show-current`
   - 결과가 `develop`이 아니면: 작업을 중단하고 "현재 브랜치가 `<브랜치>`입니다. 이 스킬은 develop → main PR 전용이므로 develop으로 체크아웃 후 다시 시도해주세요."라고 안내.

2. **원격 최신화** — `git fetch origin main develop --quiet`
   - origin/main과 origin/develop을 최신 상태로 가져옴.

3. **develop 푸시 상태 확인** — `git rev-list --count origin/develop..HEAD`
   - 0이 아니면 로컬 develop이 origin/develop보다 앞서 있다는 뜻이므로, 사용자에게 "로컬에 푸시되지 않은 커밋 N개가 있습니다. `git push` 후 다시 시도하시겠어요?"라고 묻고 사용자 확인 후 진행.

4. **변경사항 존재 확인** — `git rev-list --count origin/main..origin/develop`
   - 0이면 "develop과 main이 동일하여 PR로 보낼 변경사항이 없습니다."라고 안내하고 중단.

5. **gh CLI 인증 확인** — `gh auth status`
   - 미인증이면 사용자에게 `gh auth login`을 안내하고 중단.

## 변경사항 분석

아래 명령으로 정보를 수집하세요. 출력이 길면 핵심 파일/커밋만 발췌해 활용해도 됩니다.

```bash
git log origin/main..origin/develop --oneline
git diff origin/main..origin/develop --stat
git diff origin/main..origin/develop
```

- 커밋 메시지 prefix(`feat:`, `fix:`, `refactor:`, `chore:` 등)와 실제 diff를 함께 보고 섹션을 분류합니다.
- 신규 파일은 `git diff origin/main..origin/develop --name-status | grep '^A'`로 별도 식별합니다.

## 기존 열린 PR 확인 (생성 vs 갱신 결정)

**가장 먼저** 동일 head→base 조합으로 **OPEN 상태**인 PR이 있는지 확인합니다. 있으면 새 PR을 만들지 않고 그 PR을 갱신합니다.

```bash
gh pr list --base main --head develop --state open --json number,title,url --limit 5
```

- 결과 배열이 비어 있으면 → "신규 생성 모드"로 진행 (아래 "PR 제목 버전 결정" 섹션부터 수행).
- 결과에 PR이 있으면 → "갱신 모드"로 전환:
  - 기존 PR의 **제목은 그대로 유지**합니다 (사용자가 명시적으로 바꿔달라고 한 경우에만 변경).
  - 본문만 새 분석 결과로 덮어쓸 예정임을 사용자에게 알리고 확인을 받은 뒤, **"PR 갱신 실행"** 섹션으로 점프합니다.
  - PR 제목 버전 결정·신규 PR 생성 단계는 건너뜁니다.

> GitHub은 동일 head→base로 열린 PR을 동시에 두 개 허용하지 않으므로, 기존 PR이 있는데도 `gh pr create`를 시도하면 반드시 실패합니다. 따라서 이 확인은 선택이 아니라 필수입니다.

## PR 제목 버전 결정 (신규 생성 모드 전용)

ISO 8601 주차 기준으로 `vYYYY.WW.N` 형식을 사용합니다 (월요일 시작, 2자리 주차).

1. **오늘 날짜의 ISO 연도/주차 계산**
   - bash로 계산하려면: `date +%G.%V` (예: `2026.18`)
   - `%G`는 ISO 주차 기준 연도, `%V`는 2자리 ISO 주차 번호. **`%Y/%U/%W`는 사용 금지** (ISO 기준이 아님).

2. **동일 주차 PR 존재 여부 조회**
   ```bash
   gh pr list --state all --base main --search "v2026.18" --json title --limit 100
   ```
   - 검색 키워드의 `2026.18` 부분은 위에서 계산한 값으로 치환.
   - `in:title` 한정자는 일부 케이스에서 매칭이 누락되는 경우가 있어 사용하지 않습니다. 대신 응답을 받은 후 클라이언트 측에서 정규식 `^v2026\.18\.(\d+)`로 제목을 필터링합니다.
   - 매칭된 제목들의 N 값 중 최대값을 찾습니다.

3. **N 결정**
   - 매칭된 PR이 하나도 없으면 N = 0
   - 매칭된 PR이 있으면 N = (최대 N) + 1
   - 최종 제목: `v2026.18.0`, `v2026.18.1`, … 형태 (prefix 없음, 순수 버전 문자열)

## PR Body 포맷

체크리스트 섹션은 절대 포함하지 않습니다. 해당 없는 섹션은 생략합니다.

```markdown
## 변경 사항

### ✨ 신규 기능

**기능 제목** (`파일 경로`)
- 항목 설명
- 항목 설명

### 🐛 버그 수정

**수정 내용** (`파일 경로`)
- 항목 설명

### ♻️ 리팩토링

**리팩토링 내용** (`파일 경로`)
- 항목 설명

### 🆕 신규 파일

| 파일 | 설명 |
|---|---|
| `경로/파일명` | 한 줄 설명 |
```

### 작성 원칙

- 섹션 이모지 고정: ✨ 신규 기능 / 🐛 버그 수정 / ♻️ 리팩토링 / 🆕 신규 파일
- 파일 경로는 섹션 항목 옆에 백틱으로 명시 (여러 파일이면 대표 1개)
- 항목 설명은 한국어, 사용자가 바로 이해할 수 있는 문장
- 신규 파일 테이블은 **새로 추가된 파일만** 포함 (수정 파일 제외)
- Co-Authored-By 등 공동작업자 표기 절대 금지

## PR 생성 실행 (신규 생성 모드)

위에서 만든 제목과 본문으로 아래 명령을 실행합니다. 본문은 HEREDOC으로 전달해 따옴표/줄바꿈이 깨지지 않게 합니다.

```bash
gh pr create \
  --base main \
  --head develop \
  --title "v2026.18.0" \
  --body "$(cat <<'EOF'
## 변경 사항

### ✨ 신규 기능
...
EOF
)"
```

- `--title`은 위에서 결정한 `vYYYY.WW.N` 문자열만 사용합니다 (prefix·이모지·요약 추가 금지). 예: `v2026.18.1`
- `--head`는 명시적으로 `develop`을 지정해, 다른 브랜치에서 잘못 호출되더라도 사고를 막습니다.

## PR 갱신 실행 (갱신 모드)

기존 열린 PR이 있는 경우, 위 "기존 열린 PR 확인" 단계에서 얻은 PR 번호로 본문만 갱신합니다.

```bash
gh pr edit <PR번호> --body "$(cat <<'EOF'
## 변경 사항

### ✨ 신규 기능
...
EOF
)"
```

- **제목은 건드리지 않습니다** — `--title`을 함께 넣지 않음으로써 기존 제목을 보존합니다. 사용자가 명시적으로 제목 변경을 요구한 경우에만 `--title "..."`을 추가하세요.
- 본문은 신규 생성 모드와 동일한 포맷으로 작성합니다 (체크리스트 금지, 섹션 이모지 고정).
- 갱신은 PR description을 통째로 덮어쓰므로, 기존 description에 사용자가 손으로 추가한 내용이 있을 수 있는 경우 사용자에게 한 번 더 확인을 받으세요.

## 출력

**신규 생성 성공 시** — `gh pr create`가 출력한 PR URL을 그대로 보여주고 1~2줄 요약:

```
✅ PR 생성 완료: v2026.18.0
🔗 https://github.com/.../pull/123
```

**갱신 성공 시** — 갱신된 PR 정보를 명확히 표기:

```
✅ PR #123 본문 갱신 완료 (제목 유지: v2026.18.0)
🔗 https://github.com/.../pull/123
```

실패 시 stderr 메시지를 그대로 노출하고, 흔한 원인(권한 부족, 브랜치 미푸시 등)을 짚어 안내합니다.

## 주의사항

- 이 스킬은 **PR description만 출력하지 않습니다**. 반드시 `gh pr create`까지 실행해 실제 PR을 만듭니다. 단순 description이 필요한 경우는 `/pr-description` 스킬을 사용하세요.
- base 브랜치는 항상 `main`, head 브랜치는 항상 `develop`입니다. 다른 조합이 필요하면 사용자에게 명시적으로 확인 후 진행합니다.
- PR 생성은 외부에 공개되는 작업이므로, 변경사항 분석 결과와 최종 제목·본문을 **사용자에게 한 번 보여주고 확인을 받은 뒤 실행**합니다. 자동 모드(auto)에서는 사용자가 명시적으로 "확인 없이 바로 올려줘"라고 한 경우에만 확인 단계를 생략하세요.
